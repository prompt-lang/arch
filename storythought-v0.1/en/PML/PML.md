# PML (Prompt-Lang Markup Language)

## Overview

### Definition

PML is a **declarative language** for defining AI execution flows. Its core capabilities include:

- Describing **Workflows**
- Defining **Execution Nodes (PNode)**
- Orchestrating **LLM and non-LLM systems**
- Supporting **controllable, verifiable, recoverable** execution processes
- Integrating **Skills** and **MCP (Model Context Protocol)** external capabilities
- Providing **dual views** (machine-optimized ippl / human-readable ppl)
- Supporting **explicit node communication protocols**

### Design Goals

| Goal | Description |
|------|-------------|
| **Declarative** | Users describe "what to do," not "how to do it" |
| **Deterministic Execution** | Same input + same DAG â†’ predictable output (LLM portions affected by model randomness, but flow is deterministic) |
| **Observable** | Logs, metrics, traces, evaluations, checkpoints |
| **Composable** | PNodes can be nested into more complex PNodes; Workflows are reusable |
| **LLM-Native** | Native support for Prompt templates, model config, dynamic planning, tool calls, multi-turn dialogue |
| **Safe & Reliable** | Compensation transactions, circuit breaking, permission constraints, resource quotas, sandbox execution |
| **Protocol-Agnostic** | Supports MCP, OpenAPI, gRPC, Skills, etc. through adapters |
| **Memory-Enhanced** | Native short/long-term memory support, declarative read/write and retrieval |

### Core Concepts

| Concept | Description |
|---------|-------------|
| **Workflow** | Directed Acyclic Graph (DAG) defining node execution order and data flow |
| **PNode** | Minimum execution unit, supporting `llm`, `tool`, `function`, `planning`, `subworkflow`, `foreach`, `skill`, `mcp`, `openapi`, `grpc`, `script`, `memory`, `timer` and many other types |
| **State** | Global execution context; all values are immutable snapshots |
| **TaskGraph** | Executable graph instantiated from Workflow + PNode, divided into static, dynamic, and hybrid graphs |
| **PMLScript** | Scripts embedded in PML (JS/Python/Shell) for custom logic and CLI command generation |
| **Skill** | Reusable capability unit, sourced from MCP Tool, Subworkflow, OpenAPI, etc. |
| **IPP / PPL** | Dual views: `ippl` optimized for LLM/machine, `ppl` for human reading |
| **NNOS** | Runtime system supporting scheduling, state management, retry, validation, resource quotas, tracing, compensation, script execution, memory backends, etc. |


## Syntax Specification

### Root Structure

```yaml
pml:
  version: "1.2"

meta:
  name: string
  dialect: string
  description: string
  taskgraph: static | dynamic | hybrid
  compatible_with: [string]
  dynamic_policy: {...}
  script_engine: {...}

config:
  default_model: string
  retry: number
  timeout: number
  resources: {}
  hooks: {}
  telemetry: {}
  checkpoint: {}
  metrics: {}
  script_policy: {}
  mcp_server: {}               # Optional, expose workflow as MCP Server
  memory: {}                   # Memory backend config (v1.2+)

workflow:
  input: {}
  vars: {}
  output: {}
  nodes: []
  edges: []
  metrics: []
  evaluation: []

pnodes: []

# Top-level modules
skills: []
mcp_servers: []
scripts: []
protocols: []
command_generators: []
evaluation_experiment: {}
evaluation_templates: []
secrets: {}
permissions: {}
test: {}
```

**General Notes**

All field names are case-sensitive. Lowercase snake_case is recommended.

Data binding expressions `{{...}}` must not contain side-effect operations (e.g., function calls). Only path access and built-in filters (| length, | join, etc.) are supported.

Workflows must be Directed Acyclic Graphs (DAGs), unless `taskgraph` is declared as `dynamic` or `hybrid` with subgraphs dynamically generated by `planning` nodes.

Node IDs must be unique within a workflow and conform to identifier specifications (letters, digits, underscores; must not start with a digit).

All `output.schema` follows the JSON Schema 2020-12 draft. Runtime validates node output against this schema.


## Layered Design

### 1. Meta Layer

```
meta:
  name: "example_workflow"
  dialect: "agentic"
  description: "An adaptive data analysis workflow"
  taskgraph: "dynamic"
  compatible_with: ["1.0", "1.1", "1.2"]
  dynamic_policy:
    allowed_node_types: [llm, tool, subworkflow]
    max_total_nodes: 20
    max_depth: 3
    disallowed_tools: ["delete_database"]
  script_engine:
    default_language: "javascript"
    allowed_languages: ["javascript", "python"]
    env: { NODE_ENV: "production" }
    runtimes: { javascript: "node:18", python: "python:3.11" }
    sandbox:
      allow_network: false
      allow_filesystem: false
      memory_limit_mb: 128
      timeout_sec: 10
```

### 2. DAG Layer
```
workflow:
  input: { query: string, context: object }
  vars: 
    counter: { type: integer, default: 0 }
    conversation_history: { type: array, ttl_sec: 1800 }  # v1.2 short-term memory expiry
  output: { final_answer: "{{analyze.output.summary}}" }
  nodes: [search, analyze, report]
  edges:
    - search -> analyze
    - search -> report
    - source: search
      target: logger
      type: event
      event_name: "search_done"
  metrics:
    - name: "workflow_duration_sec"
      type: histogram
      value: "{{workflow.duration_sec}}"
  evaluation:
    - name: "final_answer_accuracy"
      type: exact_match
      reference: "{{workflow.input.expected}}"
      candidate: "{{workflow.output.final_answer}}"
      threshold: 1.0
      on_fail: "log"
```

### 3. PNode Specification

Each PNode contains the following fields (some optional):

```
id: string
type: string                     # See type list below
input: {}                         # Input mapping
process: {}                       # Execution details
output: {}                        # Output schema
control: {}                       # Retry, timeout, on_error, circuit_breaker
runtime: {}                       # Heterogeneous execution environment
compensate: string                # Compensation node ID
events: {}                        # Publish/subscribe
metrics: []                       # Node-level metrics
evaluation: []                    # Node-level evaluation
communication: {}                 # Explicit communication protocol
ippl: string | object             # Machine/LLM optimized representation
ppl: string                       # Human-readable representation
```

Supported Node Types

| Type | Description | Key Fields |
|------|-------------|------------|
| `llm` | Language model inference | model, prompt, temperature, context |
| `tool` | External tool call | tool_name, parameters |
| `function` | Deterministic function execution | code, language, sandbox |
| `planning` | Dynamic plan generation | goal, constraints, planner_config |
| `subworkflow` | Nested workflow invocation | workflow, input_mapping |
| `foreach` | Parallel iteration | items, subworkflow |
| `skill` | Reusable capability invocation | skill_id, parameters |
| `mcp` | MCP protocol tool | server, tool, parameters |
| `openapi` | REST API call | endpoint, method, headers |
| `grpc` | gRPC service call | service, method, proto |
| `script` | Inline script execution | script, language |
| `memory` | Memory read/write/search | operation, backend, query |
| `timer` | Timed trigger | delay, interval, step_freq |
| `neuro_task` | Deterministic executor (Wasm/Python) | language, code, sandbox |
| `predictor` | Non-LLM model inference | model, backend, batch_size |
| `actuator` | Physical action control | action, target_system, parameters |
| `training` | Model training task | model, dataset, learning_script |
| `rule` | Rule-based execution (RBE) | rules, conflict_resolution |
| `pipeline` | Prompt pipeline (mppl/mippl) | sequences, variable_mapping |
| `session` | Session lifecycle management | operation, session_id, context |
| `turn` | Multi-step interaction turn (MSCI) | strategy, max_turns |
| `buffer` | Prompt-Buffer management (Storyline) | operation, storyline, item_id |
| `issue_trigger` | Human decision trigger | issue, options, blocking |
| `issue_response` | Human decision response | response, on_resolve |
| `mem_view` | Memory index view | source, projection, filter |
| `markup_crud` | Markup CRUD operations | domain, operation, markup |
| `markup_query` | Markup query | domain, query, limit |
| `markup_render` | Markup rendering | render, camera, style |
| `markup_stream` | Real-time markup stream | source, window, aggregate |
| `markup_transform` | Coordinate transform | transform, method |
| `synthetic` | Synthetic prompt generation | method, sources, augmentation |

The complete PML specification supports all these node types, enabling comprehensive agent workflows from simple LLM calls to complex multi-agent orchestration with physical robot control, rule-based reasoning, self-ontology awareness, and structured human-machine collaboration through the Storyline extension.
